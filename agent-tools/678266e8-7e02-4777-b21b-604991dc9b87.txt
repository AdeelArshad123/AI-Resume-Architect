**@fernando-plank** commented · Jul 2, 2024 at 2:50pm · edited

> still have an error event update to latest version of `@supabase/ssr`.
> 
> I did a quick fix that solved the error.
> In the process of sign-in, I save a cookie reference for refresh and access tokens (sb-access-token, sb-refresh-token)
> 
> ```ts
> const cookieStore = cookies();
> 
> const { data, error } = await repository.auth.signInWithPassword({
> email,
> password,
> });
> 
> cookieStore.set('sb-access-token', data?.session?.access_token);
> cookieStore.set('sb-refresh-token', data?.session?.refresh_token);
> ```
> 
> When I try to capture user information in my API routes I've an auxiliary function:
> 
> ```ts
> retrieveAuthOrSessionUser = async () => {
> let userFinal = null;
> const cookieStore = cookies();
> const accessToken = cookieStore.get('sb-access-token');
> const refreshToken = cookieStore.get('sb-refresh-token');
> 
> const {
> data: { user },
> error,
> } = await this.supabase.auth.getUser();
> 
> if (error) {
> const { data: dataSession } = await this.supabase.auth.setSession({
> access_token: accessToken?.value ?? '',
> refresh_token: refreshToken?.value ?? '',
> });
> userFinal = dataSession?.user;
> } else {
> userFinal = user;
> }
> 
> const { id } = userFinal ?? {};
> 
> return {
> id: id ?? '',
> };
> };
> ```
> 
> Remember to erase the cookies on the process of sign-out
> 
> ```ts
> const supabaseCookies = cookies().getAll();
> 
> supabaseCookies.map((cookie) => {
> if (cookie.name.includes('sb-')) {
> cookies().delete(cookie.name);
> }
> });
> 
> ```
> 
> isn't the best scenario or code, but does the job. Hope this Helps.

**@DanielGiljam** commented · Apr 16, 2025 at 7:21am · edited

> I had this issue just now when trying to set up [Password-Based Auth](https://supabase.com/ui/docs/nextjs/password-based-auth) from the (quite recent, I think) Supabase UI Library.
> 
> After lots of debugging, I think I figured out the root cause. Not sure if anyone has mentioned it before, didn't bother reading through all the comments and nothing was particularly highlighted as a solution, so I assume it hasn't been mentioned before.
> 
> Anyways, what I found is that it came down to how browsers treat the [`Set-Cookie`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie) header in redirect responses.
> 
> When I inspected the response from reset password email link (`GET /auth/confirm?token_hash=...`), I found that in the response, the `Set-Cookie` header was set properly. So it looked something like this:
> 
> ```
> GET /auth/confirm?token_hash=...
> Host: 127.0.0.1:3000
> ...
> 
> HTTP/1.1 307 Temporary Redirect
> Location: http://localhost:3000/auth/update-password
> Set-Cookie: sb-127-auth-token=base64-...
> ...
> ```
> 
> The catch is that because the origin of the original request URL and the redirect response's [`Location`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Location) differ, the browser will ignore the `Set-Cookie` header of the redirect response.
> 
> There are cases where browsers would still respect the `Set-Cookie` header in a cross-origin redirect response, e.g. if [`SameSite`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie#samesitesamesite-value) is [`None`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie#none) and [`Secure`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Set-Cookie#secure) is set for the cookie's options. But in this case, the cookie's options (at least in development) were:
> 
> ```
> Path=/; Expires=Thu, 21 May 2026 06:26:18 GMT; Max-Age=34560000; SameSite=lax
> ```
> 
> The solution is to ensure [`SITE_URL`](https://supabase.com/docs/guides/auth/redirect-urls) always is the same origin as the URL that you pass to the `redirectTo` option in [`auth.resetPasswordForEmail`](https://supabase.com/docs/reference/javascript/auth-resetpasswordforemail) (or to the `emailRedirectTo` option in [`auth.signUp`](https://supabase.com/docs/reference/javascript/auth-signup) for that matter).
> 
> So, in short, this kind of reset password email link won't work:
> 
> ```
> ❌ won't work
> http://127.0.0.1:3001/auth/confirm?token_hash=pkce_...&type=recovery&next=http%3a%2f%2flocalhost%3a3001%2fauth%2fupdate-password
> ```
> 
> But this kind will:
> 
> ```
> ✅ will work
> http://127.0.0.1:3001/auth/confirm?token_hash=pkce_...&type=recovery&next=http%3a%2f%2f127.0.0.1%3a3001%2fauth%2fupdate-password
> ```

**@mandarini** commented · Feb 19, 2026 at 4:14pm · edited

> Hi, we think the root cause here (at least for the cases where cookies are missing on first login) has been fixed in `@supabase/ssr@0.9.0-rc.6`.
> 
> **Root cause:** When `createServerClient` was constructed, the underlying `GoTrueClient` immediately kicked off `initialize()` as a background promise. If that token refresh completed *after* the HTTP response was already generated, the `setAll` cookie handler was called too late — the cookies never made it into the response. On the next request the refresh had already happened server-side, so it appeared to "work" the second time.
> 
> **Fix:** `createServerClient` now sets `skipAutoInitialize: true`. No background init fires at construction time; session loading is deferred until your first `getSession()` / `getUser()` call, which you control within the request lifecycle.
> 
> ```bash
> npm install @supabase/ssr@0.9.0-rc.6 @supabase/supabase-js@^2.97.0
> ```
> 
> Please test and report back, we'd love to confirm this resolves it before promoting to stable.
> 
> Thank you for contributing to Supabase! 💚
